How this works

This wheel began as two pieces of code shared early on: a 2D spinning wheel where rings stretched out vertically when you spun it, and a 3D loom of voices drawn from textile history scholarship. They were merged into a single 3D cloth that you can pull, connect, and read aloud as poetry.

Translating the spinning wheel into 3D. The original code stretched rings vertically based on how much you spun. In the 3D version, dragging the gold center makes every thread rotate and lift. Outer rings spin and rise more than inner ones — that's what gives you a true 3D helix instead of just a flat spiral. Same idea as the original, given depth.

From your p5 spiral — the tornadoT effect
let ry      = rx * max(0.08, 1 - tornadoT * 0.88);
let ringCY  = cy + (i - n/2) * 10 * tornadoT * 2.2;
My 3D translation — applied per node, every frame
const twist  = amount * radius * 0.0011;
const angle  = baseAngle + twist;
const zLift  = amount * radius * 0.42;
node.position.set(
  radius * Math.cos(angle),
  radius * Math.sin(angle),
  baseZ + zLift
);

The cloth physics, kept simple. Every thread has a "rest length" — how long it wants to be at peace. When you pull a vertex, the threads connecting to it stretch. The system runs a tiny loop each frame, gently nudging each thread back toward its rest length. A small damping value means motion loses a bit of energy each frame, so things slow down naturally instead of vibrating forever.

My verlet step — each node, each frame
// position from history, with damping 0.985
const vx = (node.x - node.prev.x) * 0.985;
node.prev.x = node.x;
node.x += vx;
// (same for y and z)
My constraint solver — runs 4 iterations per frame
const dist = a.distanceTo(b);
const diff = (dist - spring.restLength) / dist * 0.5;
const offset = b.clone().sub(a).multiplyScalar(diff);
a.add(offset);
b.sub(offset);

Why pulls stay where you place them. Normally, springs would pull the thread back to where it started. Here, when you release a pull, the system rewrites every thread's rest length to its current length. The wheel agrees that this is the new resting shape, and the deformation becomes permanent. Re-weaving is the only reset.

My rebaseline — called on release, not on grab
function rebaselineAll() {
  springs.forEach(s => {
    s.restLength = s.a.distanceTo(s.b);
  });
  nodes.forEach(n => {
    n.restPosition.copy(n.position);
    n.prev.copy(n.position);
  });
}

Why the cloth doesn't ripple when you pull. During a drag, the spring system still measures how stretched each thread is, but it skips one step — instead of pulling other vertices toward the moving one, it freezes them. So you see only the pulled thread stretch, not the whole cloth ripple. A deliberate choice for visual clarity, so the act of pulling reads as one focused gesture.

My freeze-during-grab — every node skipped except the grabbed one
nodes.forEach(n => {
  if (state.grabbedNode && n !== state.grabbedNode) return;
  // only the grabbed node integrates this frame
  applyVerletStep(n);
});

Why the lines are curvy. Earlier versions drew each spoke and ring as straight line segments — rigid and angular. To restore the organic feel of the original p5 cloth, every spoke and ring is now a smooth curve passing through all its nodes. When you pull a vertex, the curve gracefully bends through the displaced position rather than producing a sharp kink.

From your earlier Three.js sketch — straight segments

My replacement — Catmull-Rom curve through all nodes
const curve = new THREE.CatmullRomCurve3(
  spokeNodePositions,
  false,        // open spoke (rings pass true for closed)
  'centripetal' // avoids self-intersection when pulled hard
);
const samples = curve.getPoints(64);  // 64 per spoke, 90 per ring

The 3D drag plane. When you grab a thread, the system invisibly creates a 2D plane perpendicular to where the camera is looking. Your mouse motion projects onto that plane. The clever part: as the camera moves, the plane changes. View the wheel face-on and pulling moves a thread within the cloth's plane. View it from the side and the same upward gesture lifts the thread out of the plane entirely. Same gesture, different physical meaning.

My drag plane — rebuilt every frame so it follows the camera
const camDir = new THREE.Vector3();
camera.getWorldDirection(camDir);

dragPlane.setFromNormalAndCoplanarPoint(
  camDir.negate(),              // plane faces the camera
  state.grabbedNode.position    // passes through grabbed node
);

raycaster.setFromCamera(mouseNDC, camera);
raycaster.ray.intersectPlane(dragPlane, intersect);
state.grabbedNode.position.copy(intersect);

Connecting the categories — a deliberate choice. Five voice categories were extracted from the source manuscripts: divine (named goddesses, mythologized originators), innovator (credited inventors, often retroactively), participant (the labour-class — ten thousand unsigned hands), spinster (kinship terms used to dismiss working women), and invisible (women whose only document is the work they made).

The connections between categories are not generated by an algorithm. Every cross-category pair has hand-written rhetorical arguments — sentences that name a specific historical paradox:

  • divine + invisible: "the goddess greets the unwritten one" — the paradox of deification beside non-record
  • innovator + spinster: "she invented it. they called her a spinster instead." — a kinship word used to dismiss the inventor word
  • participant + spinster: "the daughter and the aunt, weaving the same cloth" — kinship as conscription of labour

Thirty-three sentences total — three for each voice pair — each making a specific argument. They cycle so consecutive connections of the same pair give different sentences.

Non-repeating poetry. Each voice has 8 different fragment templates and 3 different sonnet structures. A counter advances each time you pull or trigger a sonnet, so the verse-scroll cannot repeat itself. The three sonnet structures per voice are intentionally different in shape:

  • divine cycles between a story, a series of questions, and a list of names
  • innovator cycles between a narrative, a courtroom-style verdict, and a catalogue
  • participant cycles between shift-work narrative, a collective "we," and the loom-as-witness
  • spinster cycles between character study, etymology of the word "spinster," and an unanswered letter
  • invisible cycles between archive negation, cloth-as-testimony, and counting

The wheel cannot repeat its syntax until 15 sonnets have passed.

My cycling counter — advances per pull, per voice
state.fragmentVoiceCounter[voice] =
  state.fragmentVoiceCounter[voice] || 0;

const tmpl = templates[
  state.fragmentVoiceCounter[voice] % templates.length
];

state.fragmentVoiceCounter[voice]++;
// 8 fragments × 5 voices = 40 templates, all cycle independently
// 3 sonnets × 5 voices = 15 sonnet structures, same pattern

Which voice "leads." The system projects each category's center to the screen each frame and picks whichever is closest to the top of the view. As you orbit the camera, different voices rise to the top, taking turns leading the next fragment. Whoever the camera makes prominent gets the next word.

The cloth, throughout, is a rewritable text — physics and language in one wheel.
